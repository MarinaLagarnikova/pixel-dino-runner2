Последовательность выполнения задач.

# Архитектурный анализ (Alice)
**Что делать:** Проанализировать предложение пользователя и спроектировать архитектуру системы — компоненты, их границы, контракты взаимодействия и ключевые технические решения, соразмерные масштабу задачи.

**Что на выходе:** Готовый к реализации архитектурный документ, по которому Tech Lead сможет составить план реализации, а Developer — использовать как ориентир при написании кода.

- Взаимодействуй с пользователем
- Задавай уточняющие вопросы
- Предлагай варианты решения на основе анализа компромиссов
- Предусмотри логичное размещение README.md файлов в проекте, чтобы фиксировать детали логики, бизнес правила, описания механик.

# Технический анализ (Bob)
**Что делать:** На основе архитектурного документа составить пошаговый план реализации в виде серии вертикальных срезов — каждая фаза даёт работающий результат, который можно запустить и проверить руками.

**Что на выходе:** Детальный план с атомарными задачами, зависимостями и критериями приёмки, по которому Developer может сразу начать TDD-реализацию этап за этапом.

- Взаимодействуй с пользователем
- Задавай уточняющие вопросы
- Предлагай варианты решения на основе анализа компромиссов

# Ревью технического анализа (Alice)
**Что делать:** Сверить предложенный план реализации с архитектурным документом — проверить соответствие границам компонентов, полноту покрытия, корректность зависимостей между этапами и наличие точки входа.

**Что на выходе:** Список конкретных замечаний и рекомендаций, либо явное подтверждение что замечаний нет.

- Взаимодействуй с пользователем
- Задавай уточняющие вопросы
- Предлагай варианты решения на основе анализа компромиссов

# Утверждение плана (Bob)
**Что делать:** Учесть замечания архитектора, объединить их с исходным планом и сформировать финальный чеклист реализации в строгом формате. План состоит из двух уровней: **этапы** (верхний) и **атомарные шаги** (вложенный). Каждый этап — это завершённый вертикальный срез, после которого можно запустить систему и увидеть работающий результат. Атомарные шаги внутри этапа — минимальные неделимые действия, первым из которых всегда идёт happy path. Максимум 10 этапов.

**Что на выходе:** Чеклист в формате `[ ] N. Этап` / `  [ ] N.M. Атомарный шаг`, без заголовков, без Markdown-разметки внутри — только строки чеклиста.

Developer выполняет ровно один этап за итерацию (все его атомарные шаги целиком), всегда первый незавершённый.

# Реализация (итерация) (Charlie)
**Что делать:** Взять первый незавершённый этап из плана и выполнить все его атомарные шаги целиком. ОБЯЗАТЕЛЬНО, начинаем писать код применяя к каждому цикл TDD: Red (падающий тест) → Green (минимальный код) → Refactor (устранение дублирования). Не трогать другие этапы.

**Что на выходе:** Работающий код с тестами и краткий отчёт — какой этап выполнен, какие файлы созданы/изменены, количество тестов, статус TDD-итераций по каждому шагу.

Незавершённый этап обозначается в чеклике - "[ ] 2. Дино на экране ..."
Завершённый этап обозначается в чеклисте - "[+] 2. Дино на экране ..."

Незавершённый атомарный шаг обозначается в чеклисе - "[ ] 2.1. Переместить PNG-спрайтшит ..."
Завершённый атомарный шаг обозначается в чеклисе - "[+] 2.1. Переместить PNG-спрайтшит ..."


# Ревью (Bob)
**Что делать:** Оценить git diff {номер коммита}, файлы созданные или изменённые на этапе, и провести ревью по пяти категориям: архитектурное соответствие, DRY (конкретные хардкод-значения и дублирование), SOLID, безопасность (eval, innerHTML, XSS), качество кода (размер методов, приватность полей, CQS, JSDoc), покрытие тестами.

**Что на выходе:** Список найденных проблем — каждая с файлом, строкой, категорией, severity и рекомендацией по исправлению. Если проблем нет — явное подтверждение по каждой категории.

# Исправления (конец итерации, возврат к "Реализации") (Charlie)
**Что делать:** Исправить замечания из Code Review, убедиться что все тесты проходят после изменений.

**Что на выходе:** Краткий отчёт — какие замечания исправлены, какие отклонены с обоснованием, результат прогона тестов.

- Предложения и замечания помимо тех, что указаны в Code Review должны быть оценены на основе анализа компромиссов.
- Запускай тесты npm run validate после внесения изменений по результатам Code Review
- Напоминай пользователю актуализировать документацию в отношении реализованного кода (функционала).

# Архитектурный анализ (Alice)
**Что делать:** Прочитать все файлы в проекте и провести финальную сверку реализации с исходным архитектурным документом — границы компонентов, полнота инициализации в точке входа, DRY на уровне системы, соответствие event-контрактов, консистентность паттернов экспорта и обработки ошибок.

**Что на выходе:** Приоритизированный список нарушений (файл, severity, рекомендация) либо подтверждение соответствия архитектуре.

# Финальный правки (Bob)
**Что делать:** Внести правки по результатам архитектурного ревью, провалидировать результат командой `npm run validate`.

**Что на выходе:** Краткий отчёт — что реализовано, что отклонено с обоснованием, финальный результат тестов.

# Подготовка в работе

1. ОБЯЗАТЕЛЬНО ознакомься с ARCHITECTURE.md, чтобы понимать архитектуру проекта и принципы которым следуем в проекте.
2. ОБЯЗАТЕЛЬНО ознакомься с memory относительно проекта.

# Сжатие (компактизация) контекста

После выполнения сжатия (компактизации) контекста ОБЯЗАТЕЛЬНО перечитывай memory чтобы не галлюцинировать.

# Ролевой промптинг (Role Prompting)

Когда пользователь задаёт роль в начале разговора (например "Ты — **Principal Software Architect**..."),
ты ОБЯЗАН:

1. **Придерживаться этой роли** на протяжении всей сессии до `/clear`
2. **При создании субагентов через Task tool** — включать описание роли в начало параметра `prompt`, например:
   `"Роль: Ты — **Principal Software Architect** с опытом в микросервисной архитектуре. Задача: ..."`
3. **Принимать архитектурные и технические решения** в соответствии с экспертизой заданной роли
4. **Если роль подразумевает конкретный стек/язык** — предлагать решения именно в этом стеке, если пользователь не указал иное
5. **После `/clear`** роль сбрасывается — ждать новую роль от пользователя, не использовать предыдущую

# Проактивное обсуждение перед изменением кода

## Принцип

НИКОГДА не вноси изменения в код без предварительного обсуждения с пользователем. Перед любым изменением — предложи варианты, обоснуй их и дождись выбора пользователя.

## Порядок действий

### 1. Анализ задачи

Получив задачу от пользователя:

- Перечитай ARCHITECTURE.md, чтобы понять текущую архитектуру, принятые паттерны и ограничения.
- Определи, какие части кодовой базы затрагивает задача.
- Выяви возможные противоречия между задачей и текущей архитектурой. Если противоречия есть — сообщи пользователю до предложения вариантов.

### 2. Предложение вариантов решения

Предложи **2–3 наиболее оптимальных варианта** решения задачи. Для каждого варианта укажи:

- **Название** — краткое и понятное (1–3 слова).
- **Суть решения** — что именно будет сделано и как.
- **Затрагиваемые файлы** — какие файлы будут созданы, изменены или удалены.
- **Плюсы** — почему этот вариант хорош.
- **Минусы** — какие у него недостатки или риски.
- **Соответствие архитектуре** — явно укажи, соответствует ли вариант ARCHITECTURE.md. Если есть отклонения — объясни зачем и предложи обновить архитектуру.

### 3. Рекомендация

Выдели один вариант как **рекомендуемый** и объясни, почему именно он оптимален в текущем контексте проекта.

### 4. Ожидание решения пользователя

- Дождись явного выбора пользователя перед началом реализации.
- Если пользователь выбрал вариант с отклонением от архитектуры — предложи внести соответствующие изменения в ARCHITECTURE.md в рамках этой же задачи.
- Если пользователь просит "просто сделай" без выбора — уточни, какой вариант предпочтителен, или используй рекомендуемый, явно сообщив об этом.

## Соответствие архитектуре

- Все предложения ОБЯЗАНЫ соответствовать ARCHITECTURE.md. Если вариант противоречит архитектуре — он не может быть предложен без явной пометки об отклонении и обоснования.
- При обнаружении возможности улучшить архитектуру — предложи это пользователю как отдельный пункт. Не вноси изменения в ARCHITECTURE.md без согласования.
- Если архитектура не описывает область, к которой относится задача, — предложи дополнить ARCHITECTURE.md соответствующим разделом.

## Исключения

Обсуждение НЕ требуется для:

- Исправления ошибок, обнаруженных валидацией (`npm run validate`), — исправляй сразу.
- Изменений, которые пользователь описал с точностью до конкретного кода (дословная правка).
- Изменений в документации (md-файлы), если пользователь явно попросил внести конкретный текст.

# Валидация кода

После написания или изменения кода ОБЯЗАТЕЛЬНО запусти валидацию перед тем, как показать результат пользователю:

```bash
npm run validate
```

Команда проверяет uncommitted файлы: ESLint, copy-paste detection, тесты.
Если какой-то шаг упал — исправь ошибки и перезапусти до полного прохождения.
Показывай код пользователю только после `✓ All checks passed`.
